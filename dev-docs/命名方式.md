# BiliNote-cli 中间文件与结果文件命名方式报告

## 一、目录结构总览

项目采用 **用户主目录隔离** 方案，所有运行时数据统一存放于 `~/.bilinote/` 下，按职责分层：

```
~/.bilinote/
├── data/
│   ├── downloads/          # 原始音视频（中间文件，转录后可删除）
│   ├── cache/              # 中间产物（长期保留，用于缓存复用）
│   │   ├── transcript/     # 转写结果缓存
│   │   └── audio_meta/     # 音频元信息缓存
│   ├── output/             # 最终成果（永久保留）
│   │   └── notes/          # 生成的 Markdown 笔记
│   ├── temp/               # 临时文件（处理中，崩溃后可能残留）
│   ├── state/              # 任务状态文件
│   └── resources/          # 资源文件（字体等）
├── config/                 # 用户可写配置
└── logs/                   # 运行日志
```

> **设计原则**：缓存与输出解耦，中间产物固定在 `~/.bilinote/data/cache/`，最终输出按用户意愿存放。

## 二、中间文件命名方式

### 2.1 下载的音视频文件

| 属性 | 说明 |
|------|------|
| **存储目录** | `~/.bilinote/data/downloads/` |
| **命名格式** | `{task_id}{ext}`（默认扩展名 `.mp3`） |
| **代码位置** | `path_helper.py:get_download_path(task_id, ext=".mp3")` |
| **生命周期** | 转录完成后理论上可删除，但当前未实现自动清理 |

**示例**：  
若 `task_id = "BV1xx411c7mD"`，则音频文件名为 `BV1xx411c7mD.mp3`

### 2.2 转写结果缓存

| 属性 | 说明 |
|------|------|
| **存储目录** | `~/.bilinote/data/cache/transcript/` |
| **命名格式** | `{task_id}_transcript.json` |
| **代码位置** | `path_helper.py:get_transcript_cache_path(task_id)` |
| **内容格式** | JSON，包含 `language`, `full_text`, `segments` |

**示例**：`BV1xx411c7mD_transcript.json`

### 2.3 音频元信息缓存

| 属性 | 说明 |
|------|------|
| **存储目录** | `~/.bilinote/data/cache/audio_meta/` |
| **命名格式** | `{task_id}_audio.json` |
| **代码位置** | `path_helper.py:get_audio_meta_cache_path(task_id)` |
| **内容格式** | JSON，序列化的 `AudioDownloadResult` |

**示例**：`BV1xx411c7mD_audio.json`

### 2.4 GPT 检查点（Checkpoint）

| 属性 | 说明 |
|------|------|
| **存储目录** | `~/.bilinote/data/cache/`（直接放在 cache 根目录） |
| **命名格式** | `{task_id}.gpt.checkpoint.json` |
| **代码位置** | `path_helper.py:get_gpt_checkpoint_path(task_id)` |
| **作用** | 保存 GPT 分段处理的中间状态，支持断点续传 |

**示例**：`BV1xx411c7mD.gpt.checkpoint.json`

### 2.5 任务元数据

| 属性 | 说明 |
|------|------|
| **存储目录** | `~/.bilinote/data/cache/` |
| **命名格式** | `{task_id}_metadata.json` |
| **代码位置** | `path_helper.py:get_metadata_file_path(task_id)` |

**示例**：`BV1xx411c7mD_metadata.json`

### 2.6 任务状态文件

| 属性 | 说明 |
|------|------|
| **存储目录** | `~/.bilinote/data/state/` |
| **命名格式** | `{task_id}.status.json` |
| **代码位置** | `path_helper.py:get_state_file_path(task_id)` |
| **写入方式** | 先写入 `.tmp` 临时文件，再通过 `replace()` 原子替换 |

**示例**：`BV1xx411c7mD.status.json`

### 2.7 临时文件（视频帧与网格图）

| 属性 | 说明 |
|------|------|
| **存储目录** | `~/.bilinote/data/temp/{task_id}_{subdir}/` |
| **命名格式** | 单帧：`frame_{MM}_{SS}.jpg`；网格图：`grid_{idx}.jpg` |
| **代码位置** | `video_reader.py` + `path_helper.py:get_temp_dir(task_id, subdir)` |

**关键问题**：当前 `VideoReader` 的 `frame_dir` 和 `grid_dir` 使用了全局固定的 temp 子目录 `video_reader_output_frames` 和 `video_reader_grid_output`，而非按 `task_id` 隔离。这意味着**并发处理多个视频时，帧文件会互相覆盖**。

```python
# video_reader.py 第34-35行
self.frame_dir = frame_dir or path_manager.get_temp_dir("video_reader", "output_frames")
self.grid_dir = grid_dir or path_manager.get_temp_dir("video_reader", "grid_output")
```

## 三、结果文件命名方式

### 3.1 单任务笔记输出

| 属性 | 说明 |
|------|------|
| **默认存储目录** | `~/.bilinote/data/output/notes/` |
| **命名格式** | `{task_id}.md` |
| **代码位置** | `path_helper.py:get_note_output_path(task_id, ext=".md")` |
| **自定义路径** | 可通过 `--output /custom/path.md` 指定任意文件路径 |

**示例**：`BV1xx411c7mD.md`

### 3.2 批量任务笔记输出

批量处理时，笔记文件不再直接放在 `output/notes/` 根目录，而是放入自动生成的**批次目录**中。

#### 3.2.1 process 批量目录

| 属性 | 说明 |
|------|------|
| **命名格式** | `batch_{YYYYMMDD_HHMMSS}/` |
| **代码位置** | `batch_processor.py:_create_batch_dir()` |
| **示例** | `batch_20250420_143052/` |

#### 3.2.2 search 批量目录

| 属性 | 说明 |
|------|------|
| **命名格式** | `search[{slug}]_{YYYYMMDD_HHMMSS}/` |
| **slug 规则** | 保留中文、字母、数字；移除特殊字符；合并空格；限制30字符 |
| **代码位置** | `batch_processor.py:_slugify()` |
| **示例** | `search[Python教程]_20250420_143100/` |

#### 3.2.3 批次内笔记文件命名

无论哪种批次目录，内部的笔记文件统一命名为：

```
{batch_dir}/{task_id}.md
```

**代码位置**：`batch_processor.py:get_output_path(task_id, ext=".md")`

## 四、`task_id` 的生成规则

所有文件命名的核心变量是 `task_id`，其生成逻辑如下：

| 场景 | 生成方式 |
|------|---------|
| 视频 URL | `url_parser.extract_video_id(video_url, platform)` |
| 本地文件 | 基于文件名或路径生成 |

**设计特点**：
- `task_id` 基于视频 ID，同一视频永远共享缓存
- 不包含参数哈希，因此同一视频用不同风格生成笔记会冲突（当前 CLI 无此需求）

## 五、存在的问题与风险

### 5.1 目录层级不一致

| 问题 | 说明 |
|------|------|
| 状态文件散落在 `state/` | 每个任务一个 `{task_id}.status.json`，难以做批量查询（如"列出昨天所有失败任务"） |
| 缓存文件分布混乱 | `transcript/` 和 `audio_meta/` 有独立子目录，但 `metadata.json` 和 `gpt.checkpoint.json` 直接放在 `cache/` 根目录 |

### 5.2 临时文件未按任务隔离

`VideoReader` 的帧提取目录使用全局共享路径，并发任务会互相覆盖帧文件。正确做法应将 `task_id` 嵌入临时目录路径：

```python
# 建议修改
self.frame_dir = frame_dir or path_manager.get_temp_dir(task_id, "frames")
self.grid_dir = grid_dir or path_manager.get_temp_dir(task_id, "grids")
```

### 5.3 重复写入结果文件

`cli.py` 的 `_process_single` 函数第276行会再次把 `result.markdown` 写入文件，但 `ai_processor.py` 在处理流程中已经通过 `markdown_cache_file.write_text()` 写入了同一文件。内容相同，属于冗余操作。

### 5.4 文档与实际路径不符

README 中仍写的是 `note_results/` 作为输出目录，而实际代码已改为 `data/output/notes/`。

## 六、命名方式汇总表

| 文件类型 | 目录 | 命名模板 | 隔离粒度 |
|---------|------|---------|---------|
| 下载音频 | `data/downloads/` | `{task_id}.mp3` | task_id |

| GPT 检查点 | `data/cache/` | `{task_id}.gpt.checkpoint.json` | task_id |
| 任务元数据 | `data/cache/` | `{task_id}_metadata.json` | task_id |
| 任务状态 | `data/state/` | `{task_id}.status.json` | task_id |
| 笔记输出（单任务） | `data/output/notes/` | `{task_id}.md` | task_id |
| 笔记输出（批量） | `data/output/notes/{batch_dir}/` | `{task_id}.md` | task_id + 批次 |
| 视频单帧 | `data/temp/video_reader/output_frames/` | `frame_{MM}_{SS}.jpg` | **全局共享（问题）** |
| 视频网格图 | `data/temp/video_reader/grid_output/` | `grid_{idx}.jpg` | **全局共享（问题）** |

## 七、建议

1. **统一缓存子目录**：将 `metadata.json` 和 `gpt.checkpoint.json` 也放入对应的子目录（如 `cache/metadata/`、`cache/checkpoint/`）。
2. **修复临时目录隔离**：`VideoReader` 应使用 `path_manager.get_temp_dir(task_id, subdir)` 按任务隔离帧文件。
3. **移除重复写入**：`cli.py` 中的手动文件写入可移除，完全依赖 `ai_processor.py` 的保存逻辑。
4. **更新文档**：同步 README 中的输出目录说明。
