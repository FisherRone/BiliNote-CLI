>统一的唯一的历史记录文件， 放在用户目录下。给所有类型任务提供重试服务。
这个历史记录文件采取什么格式？
如何让重试服务能够兼容单个处理和批量处理（以及未来的 AI 异步并行调用），并且具有更细粒度？

## 现有机制盘点

| 模块 | 状态/历史存储 | 重试机制 | 断点续传 |
|------|-------------|---------|---------|
| GPT (`UniversalGPT`) | `~/.bilinote/data/cache/{task_id}.gpt.checkpoint.json` | ✅ 请求级，3次指数退避 | ✅ 按 chunk 保存检查点 |
| 转录 (`transcriber`) | `~/.bilinote/data/cache/transcript/{task_id}_transcript.json` | ❌ 无 | ⚠️ 失败保留已有缓存 |
| 下载 (`downloader`) | `~/.bilinote/data/cache/audio_meta/{task_id}_audio.json` | ❌ 无 | ⚠️ 失败保留已有缓存 |
| 任务状态 (`NoteGenerator`) | `~/.bilinote/data/state/{task_id}.status.json` | ❌ 无 | ❌ 失败即 FAILED |
| 批量处理 (`BatchProcessor`) | 内存中的 `BatchResult` 列表 | ❌ 无，失败即跳过 | ❌ 无 |

当前状态是**分散的、不可查询的、无统一历史视图**的。

## 1. 历史记录文件采取什么格式？

推荐 **`~/.bilinote/bilinote.db` (SQLite，启用 WAL 模式)**。

理由：
- **零外部依赖**：`sqlite3` 是 Python 标准库。
- **事务与并发安全**：WAL 模式支持多读单写，兼容未来的异步并行 worker。
- **可索引查询**：快速筛选 `"找出 batch_xxx 中所有 FAILED 的任务"`、`"列出最近 7 天失败的任务"`。
- **单文件**：符合"统一的唯一的历史记录文件"要求，用户目录下只有一个 `.db` 文件。

如果你极度排斥 `.db`，备选是 **`~/.bilinote/history.jsonl`**（JSON Lines，每行一个不可变事件）。缺点是文件会膨胀、查询需全量扫描、并发写入需自行加锁。

### 建议的表结构（极简版）

```sql
-- 当前任务状态（替代现有分散的 *.status.json）
CREATE TABLE tasks (
    task_id      TEXT PRIMARY KEY,
    batch_id     TEXT,           -- NULL 表示单任务
    type         TEXT,           -- 'single' | 'batch_child' | 'search'
    url          TEXT,
    platform     TEXT,
    status       TEXT,           -- PENDING / DOWNLOADING / ... / SUCCESS / FAILED
    current_step TEXT,
    retry_count  INTEGER DEFAULT 0,
    max_retries  INTEGER DEFAULT 2,
    created_at   TEXT,           -- ISO8601
    updated_at   TEXT,
    completed_at TEXT,
    error_message TEXT,
    output_path  TEXT
);

-- 历史事件流水（append-only，统一历史视图）
CREATE TABLE task_events (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id    TEXT NOT NULL,
    step       TEXT,              -- PARSING / DOWNLOADING / ...
    event      TEXT,              -- 'started' | 'completed' | 'failed' | 'retrying'
    attempt    INTEGER DEFAULT 1,
    timestamp  TEXT,
    error_message TEXT,
    FOREIGN KEY (task_id) REFERENCES tasks(task_id)
);

CREATE INDEX idx_tasks_batch ON tasks(batch_id);
CREATE INDEX idx_tasks_status ON tasks(status);
CREATE INDEX idx_events_task  ON task_events(task_id);
```

## 2. 如何让重试服务兼容单处理 / 批量处理 / 未来异步并行？

核心思路：**以 `task_id` 为唯一键，以 `step` 为最小重试单元，状态机驱动，缓存即数据源。**

### 架构层级

```
┌──────────────────────────────────────────┐
│          TaskRetryService                │
│  （统一入口：重试判断、历史记录、状态流转）  │
├──────────────────────────────────────────┤
│  - 查询 tasks 表，找出可重试的 FAILED 步骤 │
│  - 根据 current_step + retry_count 决策   │
│  - 写 task_events 流水                     │
└──────────────────────────────────────────┘
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
   单任务重试    批量重试     异步 worker 拉取
   retry <id>   retry --batch  SELECT * FROM tasks
                              WHERE status='FAILED'
                              AND retry_count < max_retries
```

### 兼容三种模式的细节

| 模式 | 如何兼容 |
|------|---------|
| **单任务** | `task_id` 直接命中 `tasks` 表，读取 `current_step` 和 `retry_count`，跳过已成功步骤的缓存直接恢复。 |
| **批量处理** | `BatchProcessor` 生成一个 `batch_id`（如 `batch_20250421_143052`），子任务共享此 `batch_id`。重试时可 `retry --batch <batch_id>` 一键重跑失败项。 |
| **异步并行** | SQLite WAL 模式 + 乐观锁。worker 抢任务时执行 `UPDATE tasks SET status='PENDING', updated_at=... WHERE task_id=? AND status='FAILED'`；影响行数为 1 才算抢到。 |

### 细粒度：步骤级（Step-level）

不要替代现有的 **请求级重试**（如 GPT 的 HTTP 429 重试），而是补充 **步骤级重试**：

| 层级 | 负责什么 | 现有情况 | 是否新建 |
|------|---------|---------|---------|
| **请求级** | HTTP 超时、限流、连接错误 | GPT 已有 | 保留，不替代 |
| **步骤级** | 下载失败、转写崩溃、GPT 总结失败 | ❌ 缺口 | **这是新建重试服务的粒度** |
| **任务级** | 从头重跑整个流程 | 目前手动重新执行 CLI | 通过步骤级组合实现 |

步骤划分与现有 `TaskStatus` 对齐：

1. `PARSING` — 解析链接与平台
2. `DOWNLOADING` — 下载音视频/元信息
3. `TRANSCRIBING` — 获取平台字幕 或 音频转写
4. `SUMMARIZING` — GPT 分块总结
5. `POST_PROCESSING` — 截图、链接替换
6. `SAVING` — 写文件、写历史记录

**为什么步骤级最合理？**
- 每个步骤已经有**独立的缓存文件**（audio_meta、transcript、markdown、GPT checkpoint）。
- 重试 `DOWNLOADING` 时，不会重复 `PARSING`。
- 重试 `SUMMARIZING` 时，UniversalGPT 的 checkpoint 机制直接生效。

## 3. 新的重试服务还有必要吗？若有，粒度和侧重点？

### 结论：**有必要，但定位是"步骤级/任务级调度器"，而非替代现有请求级重试。**

现有机制的三个明显缺口：

1. **下载与转录无重试**：网络抖动或模型加载失败直接导致任务 FAILED。
2. **批量任务失败即丢弃**：`BatchProcessor` 只打印 ✗，没有重试入口。
3. **没有统一历史视图**：用户无法执行 `bilinote history` 或 `bilinote retry --last-failed`。

### 粒度：步骤级（见上）

### 侧重点

按优先级排序：

| 优先级 | 侧重点 | 说明 |
|--------|--------|------|
| P0 | **状态持久化与恢复** | 把分散的 `.status.json` 和检查点统一到 `tasks` 表，支持从任意步骤恢复。 |
| P1 | **批量失败重跑** | `bilinote retry --batch <id>` 或 `bilinote retry --failed`，这是 CLI 批量场景最痛的地方。 |
| P2 | **利用现有缓存，不重造轮子** | 重试不是清零重来，而是让 `NoteGenerator` 的各步骤方法发现缓存已存在后直接跳过。 |
| P3 | **可观测性** | `bilinote status <task_id>` 可以从 `task_events` 拉取完整时间线，而不是只看当前状态。 |

### 一个最小可行的实现思路（符合 CLAUDE.md）

不需要先造一个完美的重试框架。可以分两步：

**第一步（MVP）**：把 `tasks` 和 `task_events` 表接入现有 `NoteGenerator._update_status()`，先把历史记录统一起来。

**第二步**：给 `NoteGenerator.generate()` 加一个内部 `_run_step(step_name, fn, **kwargs)` 包装器：

```python
def _run_step(self, task_id, step_name, fn, *args, **kwargs):
    self._log_event(task_id, step_name, "started")
    try:
        result = fn(*args, **kwargs)
        self._log_event(task_id, step_name, "completed")
        return result
    except Exception as e:
        self._log_event(task_id, step_name, "failed", error=str(e))
        # 根据策略决定是否抛出让外层重试
        raise
```

然后 `BatchProcessor` 在批量结束后，提供一个 `retry_failed()` 方法，读取 `tasks` 表中 `batch_id = self.batch_id AND status = 'FAILED'` 的任务重新入队。

这样既不动现有 GPT 的请求级重试，也不破坏现有缓存逻辑，只是把**状态历史统一了、步骤边界明确了、批量重试入口打开了**。



>分析一下目前的缓存目录设计是否合理，能否兼容这个方案（假设用户指定保存的地址或批量处理的保存目录，缓存会出现变化吗）

我结合 `cli.py`、`note.py`、`path_helper.py` 的完整代码，对缓存目录设计做一次系统性分析。

## 当前缓存目录设计的核心特征

```
~/.bilinote/
├── data/
│   ├── cache/                    ← 中间产物（与 output 完全解耦）
│   │   ├── transcript/{task_id}_transcript.json
│   │   ├── audio_meta/{task_id}_audio.json
│   │   └── {task_id}.gpt.checkpoint.json
│   ├── state/{task_id}.status.json
│   ├── output/notes/{task_id}.md  ← 默认最终输出
│   ├── temp/
│   └── downloads/
```

**关键发现**：所有缓存（转写、音频元信息、GPT checkpoint、状态文件）的命名**只依赖 `task_id`**，而 `task_id = extract_video_id(video_url, platform)`。**无论用户是否指定 `--output` 或 `--output-dir`，缓存位置始终不变。**

---

## 设计合理性评估

### 合理之处

| 设计点 | 评价 |
|--------|------|
| **缓存与输出解耦** | 中间产物固定在 `~/.bilinote/data/cache/`，最终输出按用户意愿存放。符合 Unix 工具惯例，输出目录保持干净。 |
| **批量处理时缓存复用** | `BatchProcessor` 只改变 `batch_dir`（最终 `.md` 的存放位置），不改变缓存路径。同一视频在不同批次处理时，转写/音频缓存天然命中。 |
| **task_id 基于视频 ID** | 以视频 URL 的 video_id 为键，同一视频永远共享缓存，节省转写时间。 |

### 风险与问题

| 问题 | 影响 | 说明 |
|------|------|------|
| **状态文件散落在 `state/` 目录** | 与统一历史记录方案冲突 | 当前 `{task_id}.status.json` 无法做批量查询（如"列出昨天所有失败任务"），每个任务一个文件也难以支持并发写。 |
| **GPT checkpoint 的 `checkpoint_key = task_id`** | 理论上参数变化会冲突 | 实际有 `source_signature` 做 SHA256 校验（含 model、style、segments 等），参数不同会清空旧 checkpoint，所以**当前是安全的**。 |
| **task_id 不包含参数哈希** | 未来限制 | 如果以后想支持"同一个视频，分别用学术风和口语风生成两份笔记对比"，现有 `task_id`-only 的缓存键会冲突。但目前 CLI 没有此需求。 |
| **cli.py 的重复写入** | 冗余而非错误 | `_process_single` 第276行会再次把 `result.markdown` 写入文件，但 `note.py` 的 `_summarize_text` 已经写了同一个 `markdown_cache_file`。内容相同， harmless but dirty。 |

## 能否兼容"统一历史记录 + 重试服务"方案？

**结论：完全兼容，且两者是天然的互补关系。**

### 兼容性分析

```
┌─────────────────────────────────────────────────────────────┐
│                    统一历史记录 (tasks.db)                     │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────────┐  │
│  │ tasks 表    │    │ task_events │    │  不存实际缓存内容  │  │
│  │ (状态/步骤) │ +  │ (事件流水)  │    │  只存元信息与指针  │  │
│  └─────────────┘    └─────────────┘    └─────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼ 通过 task_id 关联
┌─────────────────────────────────────────────────────────────┐
│                    现有缓存文件系统（不变）                     │
│  data/cache/transcript/{task_id}_transcript.json              │
│  data/cache/audio_meta/{task_id}_audio.json                   │
│  data/cache/{task_id}.gpt.checkpoint.json                     │
└─────────────────────────────────────────────────────────────┘
```

- **历史记录存"状态和事件"**：哪个步骤失败了、重试几次、什么错误。
- **文件系统存"实际数据"**：转写文本、音频元信息、GPT 中间结果。
- **重试服务的工作流**：查 `tasks.db` 得知"task_xxx 在 DOWNLOADING 步骤失败，已重试 1 次" → 调 `NoteGenerator` 从该步骤恢复 → `note.py` 自动发现 `audio_meta` 缓存已存在则跳过 → 继续后续步骤。

### 三种场景下缓存是否会出现变化

| 场景 | 最终输出位置 | 缓存位置是否变化 | 说明 |
|------|------------|----------------|------|
| **单任务默认输出** | `~/.bilinote/data/output/notes/{task_id}.md` | ❌ 不变 | 缓存和输出都在默认目录。 |
| **单任务 `--output /custom/path.md`** | `/custom/path.md` | ❌ 不变 | 只有最终 `.md` 被写到自定义路径，转写/音频/GPT 缓存仍在 `~/.bilinote/data/cache/`。**这是正确行为**。 |
| **批量处理 `--output-dir /custom/dir`** | `/custom/dir/batch_xxx/{task_id}.md` | ❌ 不变 | 批量目录只接收最终笔记，缓存全局共享。 |

**所以你的问题的答案是：用户指定保存地址或批量处理保存目录时，缓存不会出现变化。这正是当前设计的意图。**

## 若要接入统一历史记录，缓存目录需要调整什么？

**最小改动清单：**

1. **状态存储迁移**：把 `NoteGenerator._update_status()` 从写 `data/state/{task_id}.status.json` 改为写 `tasks` 表；`show_task_status()` 从读 json 文件改为查表。
2. **历史事件追加**：在 `_update_status()` 或新的 `_run_step()` 包装器中，同时向 `task_events` 表插入事件流水。
3. **缓存文件本身不动**：`data/cache/`、`data/downloads/` 保持现状，它们作为重试时的"数据源"继续工作。
4. **批量任务关联**：`BatchProcessor` 生成 `batch_id`，子任务写入 `tasks` 表时带上 `batch_id`，支持 `retry --batch <id>`。

### 一个建议的微调（可选）

如果担心未来同一视频不同参数的任务冲突，可以在 `path_helper.py` 中预留一个**参数化缓存键**的扩展点：

```python
# path_helper.py 中
def get_transcript_cache_path(self, task_id: str, variant: str = "") -> str:
    """variant 可用于区分同一视频的不同处理参数（如不同模型、风格）"""
    name = f"{task_id}_{variant}_transcript.json" if variant else f"{task_id}_transcript.json"
    return os.path.join(self.cache_transcript_dir, name)
```

当前 `variant` 留空即可，未来需要时传入参数哈希，无需推翻现有目录结构。

## 可以直接回答：**完全可以，而且对很多场景已经足够。**

如果你只想要“任务内重试”（即单个任务执行过程中自动重试失败的步骤），完全不需要我前面设计的那些表、状态机、并发控制等复杂机制。你的需求可以用**极简方案**满足。

## 极简方案：只做任务内重试

### 实现方式

在每个步骤函数内部加一个重试循环，例如：

```python
def download_with_retry(url, max_retries=3):
    for attempt in range(1, max_retries+1):
        try:
            return downloader.download(url)
        except NetworkError as e:
            if attempt == max_retries:
                raise
            wait = 2 ** attempt  # 指数退避
            time.sleep(wait)
```

同样适用于转录、GPT 总结等步骤。你已经为 GPT 做了请求级重试，只需给下载和转录补上类似的逻辑。

### 需要改动的代码位置

- `downloader.py`：下载函数增加重试循环
- `transcriber.py`：转录函数（尤其是调用 Whisper 或 API 的部分）增加重试
- 其他步骤（如 `post_processing`）视情况

**完全不涉及数据库、状态文件、批量重试命令。**

---

## 这个方案的优缺点

| 优点 | 缺点 |
|------|------|
| 实现极简单，几十行代码 | 失败的任务无法手动重试（只能重新跑整个命令） |
| 无外部依赖，无状态管理 | 批量处理时，一个任务失败就跳过，没有后续补救 |
| 无需担心并发、锁、数据库损坏 | 没有历史记录，用户不知道之前哪些任务失败了 |
| 适合脚本式、一次性使用 | 无法实现“只重试失败的批量子任务” |
| 对网络抖动的常见情况足够 | 如果程序崩溃，已经执行的部分进度可能丢失（除非已有缓存） |

---

## 关键判断：你的典型使用场景是什么？

### 场景 A：用户每次运行一个或几个任务，失败后愿意重新运行整个命令

- **适合极简方案**。例如：用户运行 `bilinote https://youtu.be/xxx`，偶尔网络问题失败，重新运行一次即可。已有的 GPT checkpoint 和转录/下载缓存可以避免重复工作。

### 场景 B：用户经常运行大批量任务（如几十个 URL），希望失败后能单独重试失败的，而不是全部重跑

- **极简方案不够**。因为批量任务失败时，你需要记录哪些失败了，并且提供一个 `bilinote retry --failed` 命令。这时候就需要状态持久化（至少一个简单的 JSON 文件记录任务列表和状态）。

### 场景 C：用户希望查看历史失败记录，或者隔几天后重试之前失败的任务

- **极简方案不够**。因为没有持久化历史，程序退出后就丢失了失败信息。

## 折中方案：极简状态（JSON Lines）

如果你觉得 SQLite 太重，但又想支持“重试失败的任务”，可以采用 **`~/.bilinote/failed_tasks.jsonl`**：

```json
{"task_id": "abc123", "url": "https://...", "failed_step": "download", "timestamp": "2025-04-21T10:00:00"}
```

然后提供 `bilinote retry --failed` 命令读取这个文件，重新运行这些任务。这样：
- 没有复杂的表结构、并发控制
- 支持批量重试失败任务
- 实现依然简单（读写 JSON 行）

缺点：无法记录成功历史、无法按 batch 重试、查询能力弱。但对小规模使用可能足够。

> **先实现任务内重试（给 download 和 transcriber 加 retry 循环），这能解决 80% 的网络抖动问题。  
> 如果后续用户反馈需要手动重试失败任务，再加一个简单的 `failed_tasks.jsonl` 记录。  
> 只有当你明确需要并发 worker、复杂查询、历史审计时，才值得做完整的 SQLite 设计。**

你不需要现在就做那个复杂的重试服务。保持简单，按需演进。
